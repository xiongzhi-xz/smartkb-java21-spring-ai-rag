# SmartKB Advanced RAG 演示知识文档

> 本文档用于 SmartKB 本地演示和回归测试。它刻意写得比普通测试文档更长，覆盖 Java 21 Virtual Threads、Spring AI、pgvector、Advanced RAG、流式输出、引用片段和排障经验，上传后应该能切分出多个 chunk，便于验证检索、重排序和引用片段展示。

## 1. 项目背景

SmartKB 是一个面向企业知识库场景的 RAG 预研项目。它的目标不是做一个简单的聊天页面，而是把大模型能力接入到 Java 后端工程中，形成一条可运行、可调试、可演示的完整链路。用户上传文档后，系统会解析文档内容，将文本切分为多个片段，调用 Embedding 模型生成向量，再存入 PostgreSQL pgvector。用户提问时，系统会根据问题检索相关片段，把片段作为上下文交给大模型生成答案。

这个项目强调工程化落地。普通 Demo 往往只展示“问一句、答一句”，很难说明答案来自哪里，也不容易解释为什么回答可信。SmartKB 增加了文档详情查看、Advanced RAG 模式、引用片段展开和流式输出。演示时，面试官或评审可以看到原始文档、向量切片、检索来源、改写后的查询和最终答案之间的关系。

项目的本地运行方式采用混合模式。PostgreSQL 和 Redis 使用 Docker Desktop 启动，Embedding 使用本地 Ollama 的 `nomic-embed-text` 模型，Chat 模型通过中转站调用外部大模型。这样可以避免所有能力都依赖外部服务，同时保留较好的回答质量。这个取舍很适合本地开发和作品演示。

## 2. Java 21 Virtual Threads

Virtual Threads 是 Java 21 的正式特性，也是本项目的核心技术亮点之一。传统 Java Web 应用通常使用平台线程处理请求，平台线程和操作系统线程一一对应，创建和切换成本较高。当系统中存在大量阻塞 IO 时，例如数据库访问、HTTP 调用、文件读取和大模型接口调用，平台线程会长时间等待，线程池容易被耗尽。

Virtual Threads 是由 JVM 管理的轻量级线程。它让开发者继续使用同步代码风格，同时在阻塞 IO 时由 JVM 自动挂起虚拟线程，让底层平台线程去执行其他任务。这种模式特别适合 IO 密集型应用，而 RAG 系统正是典型的 IO 密集型场景。

在 SmartKB 中，Virtual Threads 可以应用在多个位置。第一是文档上传，系统需要读取文件、解析文档、调用 Embedding 服务、写入向量数据库。第二是问答链路，系统需要访问 pgvector 检索上下文，并调用大模型生成答案。第三是批量处理，如果一次上传多个文档，虚拟线程可以让每个文档处理任务独立执行，避免传统固定线程池成为瓶颈。

使用 Virtual Threads 时要注意 pinning 问题。如果在虚拟线程中长时间持有 `synchronized` 锁，虚拟线程可能固定在平台线程上，削弱并发收益。更推荐使用 `ReentrantLock` 或者减少锁粒度。对于数据库连接池，Virtual Threads 并不会让数据库连接无限增加，连接池大小仍然是实际吞吐的上限。因此，调优时不能只看线程数量，还要关注数据库连接数、外部模型接口延迟和网络带宽。

## 3. Spring Boot 与本地启动

SmartKB 基于 Spring Boot 3.3 和 Java 21。启动时建议使用 `hybrid` profile。这个 profile 下，Chat 模型走中转站，Embedding 模型走本地 Ollama。IDEA 中启动服务时，需要配置 Active profiles 为 `hybrid`，并配置 `TRANSIT_API_KEY` 和 `TRANSIT_BASE_URL` 环境变量。

本地完整启动顺序如下。第一步启动 Docker Desktop，并执行 `docker compose -f docker-compose-minimal.yml up -d`，启动 PostgreSQL 和 Redis。第二步确认 Ollama 正在运行，并已拉取 `nomic-embed-text`。第三步在 IDEA 中启动 Spring Boot 主类。第四步访问 `http://localhost:8080`，上传测试文档并开始提问。

如果浏览器访问 `http://localhost:3000` 失败，这是正常的，因为当前项目不是 Vite 或 Vue 独立前端，而是 Spring Boot 静态页面。前端文件位于 `src/main/resources/static/index.html`，实际访问地址是 `http://localhost:8080`。如果修改了静态页面但浏览器没有变化，可以重启 Spring Boot 并使用 `Ctrl + F5` 强制刷新。

## 4. 文档解析与切片

RAG 系统的质量很大程度取决于文档切片。切片太大，会导致一个 chunk 中混入太多主题，检索结果虽然命中但上下文不够精确。切片太小，又可能破坏语义完整性，大模型拿到的上下文不够回答问题。SmartKB 当前使用 Spring AI 文档读取器和文本切分能力，把文档解析为多个 `Document` 对象，并为每个对象保留元数据。

元数据至少包括 `fileName` 和 `fileType`。这些字段对 Advanced RAG 很重要，因为用户可以指定只在某个文档内检索。后续还可以扩展 `uploadTime`、`chunkIndex`、`department`、`documentVersion` 等字段。企业知识库中，按部门、版本、权限过滤非常常见，元数据过滤是 RAG 工程化不可缺少的一环。

上传短文档时，系统可能只切出一个片段。这对验证基础问答足够，但不适合演示 Advanced RAG。只有一个片段时，Top-K 检索、重排序和引用片段展开都没有明显效果。因此本文件特意写得更长，上传后应该能看到多个片段。演示时可以打开左侧文档详情，查看每个片段的内容和边界。

好的测试文档应该包含多个主题，且主题之间有一定区分度。例如本文同时包含 Virtual Threads、Spring AI、pgvector、查询改写、引用片段和性能优化。当用户分别提问这些主题时，系统应该命中不同的片段。如果所有问题都命中同一段，说明切片策略或检索参数还需要优化。

## 5. Embedding 与 pgvector

Embedding 是把文本转换为向量的过程。向量可以表达文本语义，使系统能够根据语义相似度检索文档，而不是只做关键词匹配。SmartKB 使用 Ollama 本地模型 `nomic-embed-text` 生成 Embedding。这个模型体积较小，适合本地开发和演示。

生成向量后，系统将文档片段写入 PostgreSQL 的 `vector_store` 表。pgvector 让 PostgreSQL 具备向量检索能力，可以在同一个数据库中保存业务数据、文档内容、元数据和向量。对于 Java 后端项目来说，这比单独引入专用向量数据库更容易部署，也更容易讲清楚工程取舍。

pgvector 可以配合 HNSW 索引提升检索性能。HNSW 是近似最近邻搜索算法，适合高维向量检索。实际生产中，索引参数会影响召回率和查询速度，需要根据数据规模调优。对于当前演示项目，重点是证明 PostgreSQL + pgvector 可以完成完整 RAG 闭环，而不是追求极限性能。

检索时常用参数包括 `topK` 和相似度阈值。`topK` 表示返回几个最相关片段。阈值太高可能导致找不到文档，阈值太低可能引入噪声。演示时如果回答经常说“未找到相关文档”，可以降低阈值或确认文档是否上传成功。如果返回内容与问题不相关，可以提高阈值或改进切片。

## 6. Spring AI Advisor

Spring AI Advisor 是本项目的另一个重要亮点。Advisor 可以把检索、上下文注入、对话记忆等横切逻辑从业务代码中抽离出来。普通 RAG 接口中，`QuestionAnswerAdvisor` 会自动完成向量检索，把检索到的内容注入 Prompt，再调用 Chat 模型生成答案。

多轮对话中，`MessageChatMemoryAdvisor` 用于管理历史消息。前端会保存 `conversationId`，后续追问会携带同一个 ID，后端根据这个 ID 读取上下文。这样用户可以先问“Virtual Threads 适合解决什么问题”，再追问“它和传统线程池相比呢”，系统可以理解“它”指代 Virtual Threads。

Advisor 的优点是代码整洁，业务服务不需要手动拼接所有上下文。它的限制是调试时需要理解 Advisor 链路，否则容易误以为模型直接回答了问题。SmartKB 通过测试接口、日志和引用片段展示，帮助开发者理解每一步发生了什么。

Advanced RAG 当前没有完全依赖默认 Advisor，而是显式执行查询改写、向量检索、元数据过滤、重排序和生成。这样做的好处是流程更可控，也更适合演示每一步的效果。生成阶段直接调用 ChatModel，避免再次触发普通 RAG Advisor，防止重复检索造成解释混乱。

## 7. Advanced RAG：查询改写

查询改写是 Advanced RAG 的第一步。用户的问题往往比较口语化，例如“它有什么优势”“这个怎么用”“适合什么场景”。如果直接拿这些问题做向量检索，可能无法命中正确片段。查询改写会结合历史上下文，把问题改写为更适合检索的表达。

例如用户先问“Java 21 Virtual Threads 是什么”，随后问“它对 AI 应用有什么价值”。查询改写可以把第二个问题变成“Java 21 Virtual Threads 在 AI 应用中的价值，包括文档解析、Embedding 调用和数据库访问等 IO 密集场景”。改写后的查询包含更多关键词，更容易命中相关文档片段。

查询改写本身也需要调用大模型，因此 Advanced 模式通常比普通对话慢一点。这个慢是有价值的，因为它提高了检索质量。演示时可以观察页面回答下方的“改写”字段，也可以查看后端日志中的“步骤 1: 查询改写”。如果改写结果偏离原意，说明 Prompt 需要调整。

查询改写不是越长越好。如果改写文本塞入太多概念，检索可能变得泛化，命中不够精准。理想的改写应该保留用户意图，补充必要上下文，并加入少量关键术语。企业系统中还可以根据用户身份、当前页面、所选文档范围等信息辅助改写。

## 8. Advanced RAG：元数据过滤

元数据过滤用于控制检索范围。SmartKB 前端在 Advanced 模式中提供文档下拉框，用户可以选择“全部文档”或某个已上传文档。如果选择了具体文件，前端会发送 `metadataFilter.fileName`，后端只保留元数据匹配的片段。

这个能力很实用。企业知识库中经常存在多个版本的制度文档，例如“请假制度 2024 版”和“请假制度 2025 版”。用户可能只想基于最新版本提问。如果没有元数据过滤，系统可能从旧文档中检索到过期答案。元数据过滤可以减少错误来源。

当前实现是在向量检索之后做内存过滤，适合演示和小数据量场景。更理想的生产实现是把过滤条件下推到向量数据库查询中，让数据库只返回符合条件的结果。这样可以减少无效检索，提高性能。Spring AI 和 pgvector 的过滤能力可以作为后续优化方向。

演示时可以上传两个不同主题的文档，然后在 Advanced 模式中选择其中一个文档提问。页面应该显示只来自该文档的参考来源和引用片段。如果引用片段来自其他文件，说明过滤逻辑有问题。

## 9. Advanced RAG：重排序

重排序用于优化检索结果顺序。向量检索返回的 Top-K 片段未必都是最适合生成答案的片段。某些片段语义相似但内容太短，无法提供完整答案；某些片段排名稍低但包含关键细节。重排序可以根据额外规则重新排序。

SmartKB 当前使用轻量规则版重排序，按内容长度降序排列。这个策略很简单，不是最优算法，但它可以体现重排序在流程中的位置。生产环境中可以使用专门的 rerank 模型，例如 Cohere Rerank、bge-reranker 或其他交叉编码器模型。重排序模型会同时看问题和候选片段，给出更精确的相关性评分。

重排序的成本也需要考虑。如果每次检索都把几十个片段交给 rerank 模型，会增加延迟和费用。常见做法是先用向量检索召回较多候选，再用 rerank 选出最相关的少量片段给大模型。这个流程可以平衡召回率、准确率和成本。

演示时可以强调：当前项目已经具备重排序扩展点，后续替换为专业 rerank 模型不会影响整体架构。这个说法能体现系统不是一次性 Demo，而是有演进空间的工程设计。

## 10. 流式输出与用户体验

大模型生成答案可能比较慢，尤其是通过外部中转站调用时，延迟受网络、模型排队和回答长度影响。如果前端一直空白，用户会以为系统没有工作。SmartKB 增加了等待气泡和流式输出，解决感知延迟问题。

在普通对话模式中，前端调用 `/api/chat/conversation/stream`。后端使用 SSE 推送事件，先返回 `conversationId`，再逐段推送 `token`，最后发送 `done`。前端读取 `ReadableStream`，解析 SSE 数据，把内容实时追加到同一个回答气泡中。

流式输出不能让模型本身更快，但能显著改善体验。用户看到文字逐段出现，会知道系统正在工作。对于长答案，流式输出尤其重要。演示时可以问一个需要较长解释的问题，例如“请完整说明 Virtual Threads 在 RAG 系统中的应用价值”，观察答案逐步出现。

Advanced 模式暂时仍使用完整返回，因为它需要先完成查询改写、检索、过滤、重排序和引用片段统计。后续也可以把 Advanced 模式改造成分阶段流式：先显示“查询已改写”，再显示“命中 5 个片段”，最后流式输出答案。这样可解释性会更强。

## 11. 引用片段与可解释性

RAG 系统最常见的问题是答案看起来合理，但用户不知道依据是什么。SmartKB 在 Advanced 模式中返回引用片段，前端在回答下方显示“查看引用片段”。展开后可以看到命中的文件名、chunkId 和片段内容预览。

引用片段让系统具备可解释性。用户可以检查答案是否真的来自知识库，也可以发现检索是否命中了错误片段。对于企业知识库，这一点非常重要。很多业务问题不能只靠模型自信回答，必须能追溯到制度、流程或技术文档中的具体依据。

当前引用片段只显示内容预览，长度控制在约 260 个字符，避免聊天界面过于臃肿。后续可以增加“打开完整片段”或“跳转到文档详情”的功能。文档详情窗口已经支持查看所有 chunks，未来可以根据 chunkId 定位到具体片段。

演示时可以先提问一个明确问题，然后展开引用片段，指出答案中的某个观点来自哪个 chunk。这个动作能很好地区分 SmartKB 和普通聊天机器人：普通聊天只能给答案，SmartKB 可以给证据。

## 12. 常见问题与排障

如果上传文档后列表为空，先确认 PostgreSQL 和 Redis 是否正常运行。可以访问 `/actuator/health` 查看健康状态。如果数据库连接失败，通常是 Docker Desktop 没启动，或者端口被占用。PostgreSQL 默认使用 `localhost:5432`，Redis 默认使用 `localhost:6379`。

如果上传成功但问答没有引用文档，可能是 Embedding 模型没有启动，或者向量表中没有数据。可以用文档列表接口确认是否有文档，也可以在数据库中查询 `vector_store` 表。若表里没有数据，说明上传流程没有真正写入向量库。

如果回答很慢，通常是 Chat 模型调用慢，而不是 pgvector 慢。可以先用短问题测试，例如“用 100 字说明 Virtual Threads 的优势”。也可以使用流式输出降低等待焦虑。如果 Advanced 模式慢，是因为它额外做了查询改写。

如果浏览器没有看到最新前端效果，可能是静态资源缓存。重启 Spring Boot 后使用 `Ctrl + F5` 强刷。如果仍然无效，确认访问的是 `http://localhost:8080`，不是 `localhost:3000`。当前项目没有独立前端开发服务器。

## 13. 性能观察点

SmartKB 的性能瓶颈可以拆成几个阶段观察。第一是文档解析耗时，PDF 和 Word 文档通常比 Markdown 慢。第二是 Embedding 生成耗时，取决于本地 Ollama 模型和机器性能。第三是向量写入耗时，取决于 PostgreSQL 和索引。第四是问答生成耗时，取决于外部大模型。

Virtual Threads 的收益主要体现在并发 IO 场景。如果单次请求本身很慢，Virtual Threads 不会让这次请求突然变快，但它可以让系统在多个请求同时进行时保持更好的吞吐。性能测试时应该区分单请求延迟和并发吞吐。

对于 RAG 查询，可以记录检索耗时、生成耗时和总耗时。检索耗时通常较短，如果检索很慢，需要检查索引和查询参数。生成耗时较长时，可以考虑减少上下文长度、要求模型简短回答、开启流式输出或切换更快模型。

生产级系统还需要可观测性。可以使用 Micrometer、Prometheus 和 Grafana 展示请求耗时、错误率、模型调用次数、Embedding 生成耗时和数据库连接池状态。SmartKB 已经引入 Actuator 和 Prometheus 相关依赖，后续可以继续补仪表盘。

## 14. 面试讲解建议

讲这个项目时，不要只说“我做了一个 RAG 系统”。更好的讲法是：我用 Java 21 和 Spring AI 做了一个企业知识库预研项目，重点验证了 Virtual Threads 在 AI 工程中的 IO 并发价值，使用 PostgreSQL pgvector 做向量存储，并通过 Advisor、Advanced RAG 和引用片段展示解决可维护性和可解释性问题。

可以按链路讲解：用户上传文档，后端解析并切片，调用 Ollama 生成 Embedding，写入 pgvector；用户提问时，系统检索相关片段，普通模式通过 Advisor 自动注入上下文，Advanced 模式显式执行查询改写、元数据过滤和重排序；最后模型生成答案，前端展示引用来源和具体片段。

被问到为什么不用专用向量数据库时，可以回答：本项目定位是企业 Java 后端预研，PostgreSQL 在企业环境中更常见，pgvector 可以降低部署复杂度，并且元数据和向量可以统一管理。数据规模扩大后，可以再评估 Milvus、Qdrant 或 Elasticsearch 混合检索方案。

被问到为什么用 Virtual Threads 时，可以回答：RAG 系统包含大量阻塞 IO，Virtual Threads 让我保留同步代码风格，同时提升并发处理能力。它不替代数据库连接池和外部模型限流，但能降低线程模型复杂度，是 Java 21 在 AI 应用中的典型落点。

这些问题覆盖不同章节。正常情况下，Advanced RAG 应该命中多个不同片段。展开“查看引用片段”时，应该能看到与问题主题相符的 chunk 预览。如果提问“流式输出解决什么问题”，引用片段应来自第 10 节附近；如果提问“pgvector 的好处”，引用片段应来自第 5 节附近。

## 16. 总结

这份文档的目的不是介绍某一个单点技术，而是为 SmartKB 提供一套可重复演示的知识语料。它足够长，可以触发多个 chunk；它主题明确，可以验证检索准确性；它包含工程细节，可以支持面试讲解；它覆盖排障和性能观察，可以帮助定位本地运行问题。

当 SmartKB 能稳定完成“上传本文档 -> 生成多个片段 -> 普通对话流式回答 -> Advanced RAG 指定文档回答 -> 展开引用片段”这一流程时，项目的基础演示闭环就比较完整了。后续最自然的增强方向是 Hybrid Search、引用片段跳转、Redis 持久化会话记忆、可观测性仪表盘和更系统的性能压测。
