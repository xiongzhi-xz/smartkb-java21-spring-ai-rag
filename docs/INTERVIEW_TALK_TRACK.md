# SmartKB 项目讲解与答辩提纲

这份文档用于把 SmartKB 讲成一个清楚的 Java + AI 工程项目，而不是一组页面按钮。默认只讲 RAG 知识库主线；Agent 工作台属于扩展实验，不放进 5 分钟主演示。

## 30 秒项目介绍

SmartKB 是一个基于 Java 21、Spring Boot 和 Spring AI 的企业知识库 RAG 项目。它覆盖从文档上传、解析、切片、Embedding、pgvector 检索，到普通问答、Advanced RAG、引用片段定位、Redis 多轮会话记忆和 RAG 质量评测的完整闭环。

我做这个项目的重点不是单纯调用大模型，而是把企业知识库里更关键的工程问题落下来：文档如何入库、问题如何检索到正确片段、答案如何追溯到原文、会话上下文如何持久化，以及如何用评测指标证明检索链路确实变好。

## 5 分钟演示顺序

1. 打开首页，只展示“智能问答”主线。
2. 上传 `test-docs/advanced-rag-demo.md`，说明文档会被解析、切片、生成 Embedding，并写入 PostgreSQL pgvector。
3. 打开文档详情，看 chunk 列表，说明知识库不是黑盒聊天，而是有可检查的原文片段。
4. 用普通问答提问：
   ```text
   Java 21 Virtual Threads 适合解决什么问题？
   ```
5. 继续追问，说明 Redis ChatMemory 用 conversationId 保留上下文。
6. 切换 Advanced RAG，选择指定文档，提问：
   ```text
   查询改写在 Advanced RAG 中解决什么问题？
   为什么引用片段能提升 RAG 系统可信度？
   ```
7. 展开引用片段并点击，跳转到文档详情中的具体 chunk。
8. 点击“质量评测”，展示普通向量召回、Advanced RAG、引用命中率和提升用例。

## 技术讲解主线

### 1. 为什么不是纯聊天

纯聊天页面的问题是答案不可追溯，模型可能自由发挥。SmartKB 把回答建立在文档 chunk 上，并让用户能点回原文片段，所以更适合企业知识库场景。

### 2. 文档入库链路

上传后，后端完成 UTF-8 文档解析、切片、Embedding 生成和 pgvector 入库。这里体现的是 RAG 工程基本功：数据先变成可检索的知识片段，再进入问答链路。

### 3. Advanced RAG 解决什么

Advanced RAG 不是多包一层模型调用，而是在生成答案前改善检索质量：

- 查询改写：把短问题、口语问题补全成更适合检索的表达。
- Hybrid Search：结合 pgvector 语义召回和关键词证据，降低单一路径漏召回风险。
- 元数据过滤：把检索限制在指定文档或业务范围内。
- 重排序：把更贴近问题和关键章节的片段排到前面。
- 引用片段：把最终答案和原文 chunk 连接起来。

### 4. Redis ChatMemory 的价值

Spring AI 默认内存会话重启就丢。SmartKB 自研了 Redis ChatMemory，用 Redis List 保存 conversationId 对应的消息序列，并设置 TTL。这样刷新页面或服务重启后，同一会话仍能继续追问，也更接近多实例部署时的共享会话模型。

### 5. RAG 质量评测的价值

只展示一次好看的回答不够，因为模型输出有随机性。SmartKB 的第一版 RAG Eval 只评检索和引用，不评生成文本：同一批中文问题分别看普通向量召回和 Advanced RAG 是否命中预期关键词，并统计引用片段是否覆盖证据。这样能更稳定地说明检索链路有没有变好。

## 面试高频问答

### 为什么用 pgvector，而不是单独上 Milvus 或 Elasticsearch？

这个项目目标是本地可运行、容易演示、贴近 Java 后端工程。PostgreSQL + pgvector 可以同时保存业务数据、元数据和向量，部署复杂度低；Milvus 更适合大规模向量检索，但对这个项目的演示范围偏重。Elasticsearch 适合关键词检索，向量能力也可用，但这里选择 pgvector 是为了让知识库闭环更轻量。

### 为什么需要 Hybrid Search？

单纯向量召回对语义相近的问题有效，但对术语、专有名词、章节标题可能不稳定。Hybrid Search 把语义相似度和关键词证据结合起来，能提高召回稳定性，尤其适合技术文档和企业知识库。

### 查询改写会不会把问题改偏？

会有这个风险，所以 SmartKB 保留原始问题和改写问题的双路召回，不完全依赖改写结果。演示里也把改写内容展示出来，方便排查检索是否被改偏。

### 引用片段能完全解决幻觉吗？

不能。引用片段只能降低风险，让用户可以检查答案依据。真正生产环境还需要更严格的引用约束、答案后校验、权限过滤、日志审计和人工反馈闭环。

### 为什么 RAG Eval 不直接评模型回答质量？

第一版先评检索，因为检索结果更稳定，也更容易自动化判断。模型生成文本受温度、提示词、供应商和上下文影响更大，后续可以再加入 LLM-as-judge 或人工标注评分，但不适合作为第一版核心指标。

### Java 21 Virtual Threads 在这里有什么用？

RAG 服务里有大量 IO 等待：文档读取、Embedding 调用、数据库访问、模型调用。Virtual Threads 让代码保持同步写法，同时减少平台线程阻塞浪费，适合这种 IO 密集型服务。

### 这个项目怎么扩到生产？

需要补齐权限、租户隔离、文档版本管理、异步任务队列、失败重试、权限感知检索、对象存储、审计日志、监控告警、灰度发布和更完整的 RAG Eval。当前 SmartKB 是可演示的工程闭环，不包装成生产商用系统。

### Agent 工作台要怎么讲？

只作为扩展方向，不作为主卖点。可以说仓库里保留了项目接管、任务状态、记忆层、代码上下文和 Eval 实验代码，但目前求职展示更聚焦 RAG 知识库主线，因为它更完整、更容易在短时间内看出 AI 工程价值。

## 不要过度包装

- 不说这是生产级商业知识库。
- 不说 Agent 工作台是完整多 Agent OS。
- 不说引用片段能彻底消除幻觉。
- 不说 RAG Eval 已经覆盖生成质量。
- 不提交真实 API key、token、cookie、私钥或 `.env` 内容。
