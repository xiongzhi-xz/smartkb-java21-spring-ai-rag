# SmartKB 改进与问题复盘

记录时间：2026-06-14

本文记录本阶段真正有工程价值的改进和踩坑。重点不是罗列功能，而是说明：问题如何暴露、根因是什么、怎么修、带来了什么价值、后续还能怎么演进。

## 1. 流式输出与等待反馈

### 问题表现

普通对话发出消息后，如果大模型响应慢，页面长时间没有变化。用户无法判断系统是在检索、生成，还是已经卡住。

### 根因

RAG 问答链路包含向量检索、上下文构造和外部 Chat 模型调用。真正耗时通常在模型生成阶段，尤其 Chat 走中转站时会受到网络、排队、模型速度和回答长度影响。原先前端只有完整返回，没有中间状态。

### 改进方案

- 后端新增 `/api/chat/conversation/stream`，使用 SSE 流式返回。
- SSE 事件包括 `conversation`、`token`、`done`、`error`。
- 前端使用 `ReadableStream` 解析 SSE，并将 token 逐段追加到同一个回答气泡。
- 发送后立即显示等待气泡，例如“正在结合会话上下文检索并生成回答...”。

### 工程价值

- 改善感知延迟：模型不一定更快，但用户能看到系统正在工作。
- 适合展示 RAG 系统的真实调用链路，而不是像普通请求一样黑盒等待。
- 为后续 Advanced RAG 分阶段输出打基础：查询改写完成、命中片段数、生成答案可以分阶段展示。

### 后续优化

- Advanced RAG 也可以改造成分阶段流式：先返回改写后的查询，再返回命中片段，最后流式输出答案。
- 增加请求耗时拆分：检索耗时、模型耗时、总耗时。

## 2. Advanced RAG 检索命中优化

### 问题表现

提问“查询改写在 Advanced RAG 中解决什么问题？”时，文档中明明存在对应章节，系统一开始却出现过：

- 返回 `0 个片段`。
- 回答说“文档中未找到相关信息”。
- 引用片段命中了 pgvector、重排序、推荐测试问题等泛相关片段，而不是第 7 节“Advanced RAG：查询改写”。

### 根因

原实现存在多个叠加问题：

- 只用“改写后的问题”做向量检索，改写一旦偏离原问题，就会降低召回。
- 相似度阈值固定为 `0.7`，对中文本地 embedding 和小样本文档偏严格。
- 元数据过滤是在全库 Top-K 检索之后才做，指定文档时可能先检索到其他文档，再被过滤成空。
- 重排序按内容长度降序，长片段容易压过真正相关片段。
- 关键词打分容易被“检索 / RAG / 生成”等泛词干扰，导致泛相关片段排在核心片段前面。

### 改进方案

- 双路召回：同时用原始问题和改写问题检索。
- 过滤下推：`metadataFilter.fileName` 下推到 Spring AI `SearchRequest`，让 pgvector 先限定文档范围。
- 阈值调整：候选召回阈值从 `0.7` 调整为 `0.55`，无结果时使用兜底阈值。
- 重排序优化：
  - 按问题关键词相关度排序。
  - 章节标题命中加权，例如 `## 7. Advanced RAG：查询改写`。
  - “推荐测试问题”这类只列问题、不提供正文答案的片段降权。
  - 增加领域锚点词优先级，例如 `查询改写`、`引用片段`、`元数据过滤`、`pgvector`。
- 增加 `AdvancedRagServiceTest`，覆盖“推荐问题列表”和“pgvector 泛检索片段”不能排在查询改写正文章节前面的场景。

### 工程价值

- Advanced RAG 从“能跑”提升到“可稳定演示”。
- 指定文档问答更可靠，引用片段更符合用户预期。
- 保留了后续接入专业 reranker 的扩展点。

### 当前边界

当前重排序仍是轻量规则版，不是生产级语义 rerank。后续可以替换为 bge-reranker、Cohere Rerank 或交叉编码器模型。

## 3. Markdown/TXT 中文乱码

### 问题表现

上传 `advanced-rag-demo.md` 后，文档详情和引用片段出现乱码，例如 `æ¥è¯¢æ¹å`。

### 根因

`.md` / `.txt` 曾经统一走 `TikaDocumentReader`。Tika 对部分 UTF-8 Markdown 自动探测编码不稳定，导致中文内容在入库前已经被错误解码。向量库里保存的是乱码文本，后续问答和引用片段都会继承这个问题。

### 改进方案

- `DocumentLoaderService` 中将 `.md` / `.txt` 改为直接按 UTF-8 读取。
- `.doc` / `.docx` 继续走 Tika。
- PDF 继续走 PDF Reader。
- 单测增加中文内容断言，确保 Markdown 中的“虚拟线程”等中文能被正确读取。

### 工程价值

- 文档内容、检索上下文、引用片段统一恢复为正常中文。
- 避免“看起来像前端乱码，实际是向量库旧数据”的误判。

### 注意事项

代码修复不会自动修复旧向量数据。修复前上传过的文档必须删除后重新上传。

## 4. 引用片段与可解释性

### 问题表现

初始 Advanced RAG 只返回答案和来源文件，用户无法验证答案是否真的来自文档。后来虽然有了引用片段，但排序不准时，仍会影响可信度。

### 改进方案

- `AdvancedRagResult` 增加 `references`。
- 每个引用片段包含 `fileName`、`chunkId`、`preview`。
- 前端在回答下方展示“查看引用片段”折叠区。
- 引用排序结合锚点词和章节标题优化，确保第一个片段优先成为最核心依据。

### 工程价值

- 把 RAG 从“只给答案”升级为“答案 + 证据”。
- 方便调试检索质量：用户能看到命中片段是否偏题。
- 面试讲解时能突出企业知识库的可解释性和可追溯性。

### 后续优化

- 点击引用片段后打开文档详情，并定位到对应 chunk。
- 引用片段高亮命中的关键词。

## 5. 已上传文档管理与删除

### 问题表现

修复乱码后需要删除旧文档并重新上传，但前端没有删除入口。补上删除按钮后，又遇到后端删除失败：

```text
删除文档失败: 批量删除失败: PreparedStatementCallback; bad SQL grammar [DELETE FROM vector_store WHERE id = ?]
```

### 根因

- 前端只支持查看文档详情，没有删除操作。
- 后端删除复用了 Spring AI `vectorStore.delete()`，在当前 Spring AI M1 版本和 pgvector 表结构下，内部 SQL/JDBC 参数处理不稳定。

### 改进方案

- 前端文档列表增加删除按钮，点击后确认删除。
- 删除成功后刷新文档列表。
- 如果删除的是当前 Advanced 过滤文档，自动清空过滤条件。
- 后端改为项目可控的 SQL 删除：

```sql
DELETE FROM vector_store WHERE id::text = ?
```

### 工程价值

- 本地演示和回归测试更顺畅，旧数据可以直接在 UI 中清理。
- 避免依赖 Spring AI 早期版本中不稳定的删除实现。
- 删除逻辑更容易解释和排查。

## 6. 测试语料与演示闭环

### 问题表现

早期只上传较小文档时，系统只切出 1 个 chunk。这样基础问答能跑，但无法有效展示 Top-K、重排序、引用片段、文档过滤等 Advanced RAG 能力。

### 改进方案

- 新增 `test-docs/advanced-rag-demo.md`，覆盖多个主题：
  - Java 21 Virtual Threads
  - Spring AI Advisor
  - pgvector
  - 查询改写
  - 元数据过滤
  - 重排序
  - 流式输出
  - 引用片段
  - 排障和面试讲解
- 文档上传后约 10 个 chunk，适合验证不同问题命中不同章节。

### 工程价值

- 让项目从“功能能跑”变成“可以稳定演示”。
- 更容易发现检索排序、切片边界、引用片段等真实问题。
- 可作为后续回归测试语料。

## 7. 本地运行与配置问题

### 遇到的问题

- IDEA 中把 `SPRING_PROFILES_ACTIVE=hybrid;TRANSIT_API_KEY=...` 整串填进 Active profiles，导致 Spring Boot 配置路径解析异常。
- 访问 `http://localhost:3000` 失败，以为前端没启动。
- `8080` 端口被占用，导致 Web server failed to start。

### 经验沉淀

- IDEA 的 Active profiles 只填 `hybrid`。
- API Key、Base URL 放 Environment variables。
- SmartKB 当前主页面是 Spring Boot 静态页：`http://localhost:8080`。
- `3000` 不是主页面，除非启动完整 Grafana。
- 每次后端 Java 代码修改后必须重启 Spring Boot；仅 `Ctrl + F5` 只能刷新前端静态资源缓存。

## 8. 当前剩余有价值的改进方向

- Hybrid Search：向量检索 + 关键词检索融合，减少 embedding 召回不稳定。
- 引用片段跳转：点击引用片段定位到文档详情中的具体 chunk。
- Advanced RAG 分阶段输出：显示查询改写、命中片段、生成答案的阶段状态。
- Redis 会话记忆持久化：让多轮对话状态跨进程、跨重启保存。
- 可观测性：记录检索耗时、生成耗时、模型调用次数、错误率。
- 性能压测：证明 Virtual Threads 在批量上传、并发问答等 IO 密集场景中的价值。
