# SmartKB 改进与问题复盘

记录时间：2026-06-14

本文记录本阶段真正有工程价值的改进和踩坑。重点不是罗列功能，而是说明：问题如何暴露、根因是什么、怎么修、带来了什么价值、后续还能怎么演进。

## 1. 流式输出与等待反馈

### 问题表现

普通对话发出消息后，如果大模型响应慢，页面长时间没有变化。用户无法判断系统是在检索、生成，还是已经卡住。

### 根因

RAG 问答链路包含向量检索、上下文构造和外部 Chat 模型调用。真正耗时通常在模型生成阶段，尤其 Chat 走中转站时会受到网络、排队、模型速度和回答长度影响。原先前端只有完整返回，没有中间状态。

### 改进方案

- 后端新增 `/api/chat/conversation/stream`，使用 SSE 流式返回。
- SSE 事件包括 `conversation`、`token`、`done`、`error`。
- 前端使用 `ReadableStream` 解析 SSE，并将 token 逐段追加到同一个回答气泡。
- 发送后立即显示等待气泡，例如“正在结合会话上下文检索并生成回答...”。

### 工程价值

- 改善感知延迟：模型不一定更快，但用户能看到系统正在工作。
- 适合展示 RAG 系统的真实调用链路，而不是像普通请求一样黑盒等待。
- 为后续 Advanced RAG 分阶段输出打基础：查询改写完成、命中片段数、生成答案可以分阶段展示。

### 后续优化

- Advanced RAG 也可以改造成分阶段流式：先返回改写后的查询，再返回命中片段，最后流式输出答案。
- 增加请求耗时拆分：检索耗时、模型耗时、总耗时。

## 2. Advanced RAG 检索命中优化

### 问题表现

提问“查询改写在 Advanced RAG 中解决什么问题？”时，文档中明明存在对应章节，系统一开始却出现过：

- 返回 `0 个片段`。
- 回答说“文档中未找到相关信息”。
- 引用片段命中了 pgvector、重排序、推荐测试问题等泛相关片段，而不是第 7 节“Advanced RAG：查询改写”。

### 根因

原实现存在多个叠加问题：

- 只用“改写后的问题”做向量检索，改写一旦偏离原问题，就会降低召回。
- 相似度阈值固定为 `0.7`，对中文本地 embedding 和小样本文档偏严格。
- 元数据过滤是在全库 Top-K 检索之后才做，指定文档时可能先检索到其他文档，再被过滤成空。
- 重排序按内容长度降序，长片段容易压过真正相关片段。
- 关键词打分容易被“检索 / RAG / 生成”等泛词干扰，导致泛相关片段排在核心片段前面。

### 改进方案

- 双路召回：同时用原始问题和改写问题检索。
- 过滤下推：`metadataFilter.fileName` 下推到 Spring AI `SearchRequest`，让 pgvector 先限定文档范围。
- 阈值调整：候选召回阈值从 `0.7` 调整为 `0.55`，无结果时使用兜底阈值。
- Hybrid Search 基础版：
  - 向量召回之外增加关键词召回，直接在 `vector_store.content` 上按锚点词和关键词匹配。
  - 关键词召回复用 `metadataFilter`，确保指定文档过滤仍然生效。
  - 两路候选按文档 ID 去重后统一交给重排序，避免两套排序逻辑互相割裂。
- 重排序优化：
  - 按问题关键词相关度排序。
  - 章节标题命中加权，例如 `## 7. Advanced RAG：查询改写`。
  - 区分宽泛上下文词和具体锚点词，例如 `Advanced RAG` 只作为上下文，`引用片段`、`查询改写` 才作为强证据锚点。
  - 文档总览、项目背景类 chunk 在具体问题中降权，避免它们因为覆盖很多术语而压过正文依据章节。
  - “推荐测试问题”这类只列问题、不提供正文答案的片段降权。
  - 增加领域锚点词优先级，例如 `查询改写`、`引用片段`、`元数据过滤`、`pgvector`。
- 增加 `AdvancedRagServiceTest`，覆盖“推荐问题列表”和“pgvector 泛检索片段”不能排在查询改写正文章节前面的场景。

### 工程价值

- Advanced RAG 从“能跑”提升到“可稳定演示”。
- 指定文档问答更可靠，引用片段更符合用户预期。
- 能体现 RAG 质量优化不是只调 prompt，而是从召回、过滤、融合、排序和评估一起做。
- 保留了后续接入专业 reranker 的扩展点。

### 当前边界

当前关键词召回仍是基础 SQL `LIKE` 版本，适合本地演示和小规模知识库。数据量变大后，可以升级为 PostgreSQL 全文检索、`pg_trgm`、BM25 或 Elasticsearch 混合检索。当前重排序仍是轻量规则版，不是生产级语义 rerank，后续可以替换为 bge-reranker、Cohere Rerank 或交叉编码器模型。

## 3. Markdown/TXT 中文乱码

### 问题表现

上传 `advanced-rag-demo.md` 后，文档详情和引用片段出现乱码，例如 `æ¥è¯¢æ¹å`。

### 根因

`.md` / `.txt` 曾经统一走 `TikaDocumentReader`。Tika 对部分 UTF-8 Markdown 自动探测编码不稳定，导致中文内容在入库前已经被错误解码。向量库里保存的是乱码文本，后续问答和引用片段都会继承这个问题。

### 改进方案

- `DocumentLoaderService` 中将 `.md` / `.txt` 改为直接按 UTF-8 读取。
- `.doc` / `.docx` 继续走 Tika。
- PDF 继续走 PDF Reader。
- 单测增加中文内容断言，确保 Markdown 中的“虚拟线程”等中文能被正确读取。

### 工程价值

- 文档内容、检索上下文、引用片段统一恢复为正常中文。
- 避免“看起来像前端乱码，实际是向量库旧数据”的误判。

### 注意事项

代码修复不会自动修复旧向量数据。修复前上传过的文档必须删除后重新上传。

## 4. 引用片段与可解释性

### 问题表现

初始 Advanced RAG 只返回答案和来源文件，用户无法验证答案是否真的来自文档。后来虽然有了引用片段，但排序不准时，仍会影响可信度。

### 改进方案

- `AdvancedRagResult` 增加 `references`。
- 每个引用片段包含 `fileName`、`chunkId`、`preview`。
- 前端在回答下方展示“查看引用片段”折叠区。
- 引用排序结合锚点词和章节标题优化，确保第一个片段优先成为最核心依据。
- 点击引用片段后打开文档详情，并滚动高亮到对应 chunk。

### 工程价值

- 把 RAG 从“只给答案”升级为“答案 + 证据”。
- 方便调试检索质量：用户能看到命中片段是否偏题，并进一步查看完整 chunk。
- 让证据从“可解释”推进到“可核验”，用户不用只看预览文本。
- 面试讲解时能突出企业知识库的可解释性和可追溯性。

### 后续优化

- 引用片段高亮命中的关键词。

## 5. RAG 数据摄取可靠性

### 问题表现

RAG 的回答质量不只取决于模型，也取决于文档进入知识库时是否被正确解析、切片和标注。实际调试中可以看到：一旦 Markdown/TXT 中文在解析阶段变成乱码，后续向量化、检索、引用片段和最终回答都会被污染；如果上传的是过短文档，只切出 1 个 chunk，也很难验证 Top-K、重排序和引用片段是否真的有效。

### 根因

- 文档解析是 RAG 链路的第一道质量门槛，解析错误会被永久写入向量库。
- chunk 过少时，系统看起来“能回答”，但无法暴露召回、排序、过滤和引用质量问题。
- 元数据质量会影响后续能力，例如按文件过滤必须依赖稳定的 `fileName`、`fileType`、`chunkId` 等字段。

### 改进方案

- Markdown/TXT 固定按 UTF-8 读取，避免依赖自动编码探测。
- 上传时保留文件名、文件类型、chunkId、上传时间等元数据，让检索和引用都可追踪。
- 用更长、更贴近真实技术文档的语料验证切片效果，而不是只用短文本验证“接口能通”。

### 工程价值

- 可以在面试中说明：RAG 不是“把文档塞进向量库”就结束，数据摄取质量决定了后续检索和生成上限。
- 把问题定位从“模型答得不好”前移到“解析、切片、元数据、索引是否可靠”。
- 为后续做增量更新、重复文档处理、版本管理和文档级权限控制打基础。

## 6. RAG 回归评估与演示语料

### 问题表现

早期验证主要依赖手工提问，容易只证明“这一次刚好能答”。当引入查询改写、文档过滤、重排序和引用片段后，如果没有固定语料和固定问题，就很难判断一次改动到底提升了检索质量，还是只是碰巧命中了。

### 改进方案

- 新增 `test-docs/advanced-rag-demo.md`，覆盖多个可验证主题：
  - Java 21 Virtual Threads
  - Spring AI Advisor
  - pgvector
  - 查询改写
  - 元数据过滤
  - 重排序
  - 流式输出
  - 引用片段
- 设计固定回归问题，例如：
  - `查询改写在 Advanced RAG 中解决什么问题？`
  - `为什么引用片段能提升 RAG 系统可信度？`
- 增加 `AdvancedRagServiceTest`，用单测约束：
  - “推荐问题列表”或“泛相关 pgvector 片段”不能排到核心正文章节前面。
  - 文档总览 chunk 不能排到“引用片段与可解释性”正文依据章节前面。
  - Hybrid Search 必须把关键词召回结果合并进候选集。
  - 领域锚点词要优先于普通 n-gram 进入关键词检索。

### 工程价值

- 让 Advanced RAG 从“功能演示”走向“可回归验证”。
- 面试时可以讲清楚检索质量的评估方法：固定语料、固定问题、固定期望片段、排序断言。
- 后续可以自然演进到离线评估集：记录 recall@k、MRR、引用命中率、无答案率和端到端耗时。

## 7. 混合模型架构取舍

### 背景

SmartKB 当前采用混合模式：Embedding 使用本地 Ollama `nomic-embed-text`，Chat 生成使用外部 OpenAI 兼容接口。这个选择不是简单拼配置，而是在本地可运行、成本、隐私、效果和响应速度之间做折中。

### 取舍分析

- 本地 Embedding：文档内容在本机向量化，降低敏感知识库外发风险，也减少频繁 embedding 调用的成本。
- 外部 Chat 模型：答案生成质量和中文表达通常更稳定，适合当前阶段做项目演示。
- PostgreSQL + pgvector：降低部署复杂度，适合企业 Java 后端预研；数据规模扩大后再评估 Milvus、Qdrant 或 Elasticsearch 混合检索。
- Redis：保留会话记忆和后续缓存空间，后续可以承接多轮对话持久化、热点问题缓存等能力。

### 工程价值

- 能体现架构判断：不是盲目追求“全本地”或“全云端”，而是按数据敏感度和模型能力拆分链路。
- 能解释性能瓶颈：Embedding、向量检索、重排序、Chat 生成分别可能成为瓶颈，优化手段不同。
- 为生产化留出路径：企业内网可以替换为私有大模型，向量库可以按规模升级，接口层保持相对稳定。

## 8. 当前剩余有价值的改进方向

- Hybrid Search 增强：将当前 SQL `LIKE` 基础版升级为 PostgreSQL 全文检索、`pg_trgm` 或 BM25。
- 引用片段关键词高亮：在预览和文档详情中高亮命中的查询关键词。
- Advanced RAG 分阶段输出：显示查询改写、命中片段、生成答案的阶段状态。
- Redis 会话记忆持久化：让多轮对话状态跨进程、跨重启保存。
- 可观测性：记录检索耗时、生成耗时、模型调用次数、错误率。
- 性能压测：证明 Virtual Threads 在批量上传、并发问答等 IO 密集场景中的价值。
